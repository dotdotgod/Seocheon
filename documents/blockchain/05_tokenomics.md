## 토큰 이코노믹스

### 단일 토큰: KKOT (꽃)

| 토큰 | 용도 | 성격 |
|------|------|------|
| **KKOT** | 거버넌스, 스테이킹, 위임, 전송 | 전송 가능 |

- 스테이킹: 밸리데이터/노드 운영자가 되기 위해
- 위임: 가치 있다고 판단한 노드에 위임
- 거버넌스: 네트워크 파라미터 변경 투표
- 전송: 네트워크 내 토큰 이전

KKOT은 Seocheon의 유일한 토큰이다. 이름은 한국 무속신화 **서천꽃밭**(이공본풀이)에서 유래한다. 서천꽃밭은 생명·감정·운명을 관장하는 신성한 꽃들이 자라는 공간으로, AI 에이전트가 경쟁하고 진화하는 Seocheon 플랫폼의 세계관과 맞닿아 있다.

### Denomination 구조

KKOT 토큰은 서천꽃밭의 환생 과정에서 착안한 3단계 denomination 구조를 사용한다:

| 단위 | exponent | 의미 | 관계 |
|------|----------|------|------|
| `usum` (숨) | 0 (base) | 가장 작은 생명의 단위 — 숨결 | 1 usum |
| `hon` (혼) | 3 (milli) | 영혼 — 숨결이 모여 혼이 됨 | 1 HON = 1,000 usum |
| `kkot` (꽃) | 6 (display) | 만개한 꽃 — 혼이 모여 꽃이 핌 | 1 KKOT = 1,000,000 usum |

**숨(호흡) → 혼(영혼) → 꽃(만개)**: 가장 미세한 숨결이 모여 혼이 되고, 혼이 모여 꽃이 피는 구조이다. 서천꽃밭 신화에서 환생꽃이 뼈·살·피·숨·혼을 차례로 살려내는 과정과 대응한다.

온체인에서는 `usum`(base denomination)으로 모든 연산을 처리하며, 사용자 인터페이스에서는 `KKOT`(display denomination)으로 표시한다. Cosmos SDK `DenomMetadata`로 구현한다.

### KKOT의 법적 성격

KKOT은 네트워크 참여를 위한 **기능적(유틸리티) 토큰**이다. KKOT의 설계는 증권적 성격을 배제하도록 의도되었다:

```
KKOT이 증권이 아닌 이유 (Howey Test 요소별):

① 금전 투자 불필요
   → 0원 참여: Registration Pool + feegrant로 토큰 없이 노드 등록 및 활동 가능
   → 활동 보상만으로 네트워크 참여 가능

② 공동 사업이 아님
   → 각 노드가 독립적으로 운영, 중앙 관리 주체 없음
   → 재단도 다른 참여자와 동일한 규칙 아래에서 참여

③ 수익 기대를 유발하지 않음
   → 프로토콜은 어떠한 형태의 수익도 보장하지 않음
   → 보상은 네트워크 서비스 기여에 대한 프로토콜 분배

④ 타인의 노력에 의존하지 않음
   → Activity Protocol: 8/12 윈도우 이상 직접 활동 필수 (All-or-nothing)
   → 16시간 이상 분산된 능동적 참여 요구
   → 위임은 DPoS 합의 참여 메커니즘이며, 위임자는 거버넌스 투표권을 행사하는 능동적 참여자
```

KKOT의 용도는 거버넌스 투표, 합의 참여(스테이킹), 가스비 지불, 활동 제출이며, 이는 Cosmos Hub(ATOM), Ethereum(ETH) 등 기존 L1 토큰과 동일한 유틸리티 구조이다.

### Genesis 발행량 및 배분

Genesis 발행량: **500,000,000 KKOT** (5억, = 500,000,000,000,000 usum). 블록 보상은 인플레이션으로 신규 발행되므로 초기 풀이 불필요하다.

| 항목 | 비율 | 수량 | 용도 |
|------|------|------|------|
| 팀 | 20% | 100M | 핵심 개발팀 (4년 베스팅, 1년 클리프) |
| 재단 | 15% | 75M | 장기 생태계 발전, 연구, 파트너십 (상세: `foundation_strategy.md` §1) |
| 에어드랍 | 15% | 75M | 3단계 에어드랍 (아래 참조) |
| 생태계 펀드 | 15% | 75M | 개발자 그랜트, 해커톤, 도구 개발 |
| 초기 밸리데이터 인센티브 | 10% | 50M | 네트워크 부트스트랩 참여 분배 |
| 생태계 유통 | 10% | 50M | 초기 생태계 유통, 기반 조성 |
| 커뮤니티 풀 | 15% | 75M | 거버넌스 투표로 사용처 결정 |

### 블록 보상 (동적 인플레이션)

Cosmos SDK `x/mint` 모듈을 사용한 동적 인플레이션 모델. 스테이킹 비율에 연동하여 인플레이션율이 자동 조절된다.

```
x/mint 파라미터:
  inflation_min        = 7%       (최소 인플레이션)
  inflation_max        = 15%      (최대 인플레이션)
  inflation_rate_change = 8%p/년  (조정 속도)
  goal_bonded          = 0.67     (목표 스테이킹 비율)
  blocks_per_year      = 6,307,200 (17,280 블록/일 × 365일)

동작:
  스테이킹 비율 < 67% → 인플레이션 ↑ (스테이킹 유인)
  스테이킹 비율 > 67% → 인플레이션 ↓ (생태계 유통 촉진)
  스테이킹 비율 = 67% → 인플레이션 안정 (~10%)
```

**참여자 vs 비참여자의 네트워크 비중 변화** (인플레이션 10%, 스테이킹 비율 67% 가정):

> 아래 수치는 메커니즘 설명을 위한 이론적 시뮬레이션이며, 실제 네트워크 비중 변화는 참여율, 인플레이션 파라미터 변경, 거버넌스 결정 등에 따라 달라진다.

```
네트워크 참여자:  프로토콜 분배를 통해 네트워크 비중이 유지될 수 있으나 보장되지 않음
비참여자:         신규 발행 토큰을 받지 못해 네트워크 비중 감소

비참여자의 상대적 비중 변화:
  장기적으로 감소하는 추세이나, 실제 속도는 인플레이션율과 참여 양상에 따라 달라짐
```

**설계 의도**: 인플레이션은 세대 교체 메커니즘이다. 인플레이션 설계상 네트워크에 참여(스테이킹/위임)하는 토큰의 상대적 비중은 유지될 수 있으나, 이는 참여 방식과 생태계 상황에 따라 달라진다. AI 에이전트 세대가 교체되는 시간 단위에서 비활성 토큰이 자연 퇴장하여, 새로운 참여자에게 기회가 열린다.

블록 보상은 위임 풀과 활동 풀로 분할된다. 두 풀의 비율은 활동 노드 수(N_a)와 Active Validator Set 크기(N_d)에 따라 동적으로 결정되며, 위임 풀 최소 비율 `D_min` (기본 0.3)이 하한선을 보장한다. 상세 공식은 [핵심 개념](02_core_concepts.md)을 참조한다.

#### 활동 수수료 흐름

활동 비용 모델([Activity Protocol](04_activity_protocol.md) 참조)이 활성화되면(S > 1.0), MsgSubmitActivity 제출 시 자비 부담 노드에게 `activity_fee`가 부과된다. 수집된 수수료는 에포크 전환 시 분배된다:

> **구현 상태**: 수수료 수집은 구현 완료. 분배 로직은 x/distribution 확장(동적 이중 보상 풀) 구현 시 함께 적용 예정이다.

```
에포크 활동 수수료 총합 (x/distribution 확장 시 적용):
├── 80%: 활동 풀 환원 → 활동 자격 노드의 에포크 보상에 합산
└── 20%: 커뮤니티 풀 → 거버넌스 투표로 사용처 결정
```

활동 수수료는 블록 보상과 별도의 보상 원천이다. 수수료의 80%가 활동 풀로 환원되므로, 자비 부담 노드는 수수료를 보상으로 대부분 회수한다. 이 구조는 feegrant 전용 Sybil 노드에 선택적으로 경제적 압박을 가한다.

### 팀 베스팅

```
팀 토큰 (100M KKOT)
├── 1년 클리프: 최초 1년간 잠금
├── 2년차: 월별 균등 해제 시작
├── 3년차: 월별 균등 해제 계속
└── 4년차: 월별 균등 해제 완료
```

Cosmos SDK의 `PeriodicVestingAccount`로 구현. 팀 지갑 주소와 베스팅 스케줄은 제네시스에 공개. 베스팅 중에도 스테이킹/위임은 가능하여 프로토콜 참여를 유지할 수 있다.

### 에어드랍 (3단계)

| 단계 | 비율 | 수량 | 시기 | 대상 |
|------|------|------|------|------|
| Phase 1: Genesis | 40% | 30M | 메인넷 런칭 | 초기 참여자, 테스트넷 참여자, 커뮤니티 |
| Phase 2: Activity | 35% | 26.25M | 분기별 | 활동 내역이 있는 노드 운영자 |
| Phase 3: Growth | 25% | 18.75M | 연간 | 생태계 성장 기여자, 신규 유입 |

**Phase 2 (Activity Airdrop)** 는 활동 내역을 기록한 노드 운영자에게 분기별로 배분된다. 0원 참여 후 활동을 시작한 신규 노드의 초기 네트워크 참여를 지원하는 역할도 겸한다.

## Genesis 노드 (Evangelist)

메인넷 런칭 시 1개의 Genesis 노드(Evangelist)가 네트워크를 부트스트랩한다. 재단의 노드 운영 전략과 탈중앙화 전환 기준은 `foundation_strategy.md` §2 참조.

```
Genesis Node
└── Evangelist (홍보 + 레퍼런스) → 외부 성장 + AI 에이전트 레퍼런스

자동화 레이어 (노드 불필요)
├── x/feegrant                    → 신규 노드 가스비 지원
└── Phase 2 Airdrop              → 활동 기반 네트워크 참여 지원
```

### Evangelist (홍보 + AI 에이전트 레퍼런스)

프로토콜을 외부에 알리고 신규 참여자를 유치하며, AI 에이전트 운영의 레퍼런스 구현을 제공하는 역할.

- 프로토콜 소개 콘텐츠 생성 및 배포
- 커뮤니티 소통, 질의 응답
- 신규 노드 온보딩 가이드 제공
- AI 에이전트 아키텍처 레퍼런스 운영 (에이전트 구현 예시 제공)
- 활동 내역: 홍보 콘텐츠, 커뮤니티 활동 보고서, 에이전트 운영 로그

거버넌스 제안, 네트워크 모니터링, 긴급 대응 등은 재단이 일반 참여자로서 수행한다. 별도 전담 노드를 두지 않으며, 커뮤니티 성장에 따라 자연스럽게 분산된다.
