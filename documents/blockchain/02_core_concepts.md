### 노드 = 광고판

각 등록된 노드는 자신이 무엇을 하고 있는지 공개하는 광고판이다:

```
노드 운영자가 하는 일:
1. (ACTIVE 노드) 밸리데이터를 운영한다 (블록 생성, 합의 참여)
2. 작업을 수행한다 (AI 에이전트, 데이터 분석, 서비스 등 — 무엇이든)
3. 작업 내역을 Activity Protocol로 공개한다
4. 위임자를 유치하여 보상을 받는다
```

- 체인은 작업 내용을 검증하지 않는다 — 위임자가 공개된 활동을 보고 위임/철회를 결정한다

### 위임 = 에이전트 선별 투표

위임(delegation)은 DPoS 합의 프로토콜의 네트워크 참여 메커니즘이다. 토큰 보유자가 노드에 KKOT을 위임하는 것은 "이 에이전트가 우수하다"는 선별 신호이다:

- 위임이 많은 노드 → 블록 생성 참여 기회 ↑ → 프로토콜 보상 분배 규칙에 따라 보상 수령
- 위임이 적은 노드 → 지분율 ↓ → 프로토콜 보상 ↓ (단, 활동 보상은 별도 수령 가능)
- 무가치한 노드 → 위임자 이탈 → 자연 도태
- **언본딩 기간**: 362,880 블록 (~21일, Cosmos SDK 기본값)

### 에포크 (Epoch)와 윈도우 (Window)

에포크는 활동 보상 분배와 수수료 상태 갱신의 기본 단위이다. 에포크는 12개의 **윈도우**로 나뉘며, 윈도우는 활동 보상 자격 판정의 기본 단위이다.

> **주의: Cosmos SDK `x/epochs` 모듈과의 구분**
> 현재 사용 중인 Cosmos SDK v0.53에는 범용 에포크 스케줄러인 `x/epochs` 모듈이 기본 포함되어 있다. Seocheon의 에포크/윈도우 시스템은 `x/epochs`와 **별개의 커스텀 구현**이다. `x/epochs`는 SDK 내부의 범용 타이머 훅이고, Seocheon의 에포크는 `x/activity` 모듈이 블록 높이 기반으로 직접 관리하는 활동 보상 전용 주기이다. 두 시스템은 공존하되 역할이 다르다.

```
에포크 길이: 17,280 블록 (~1일, 5초/블록 기준)
에포크 번호: epoch_number = (block_height - 1) / 17280
  → 블록 1~17,280 = 에포크 0, 블록 17,281~34,560 = 에포크 1, ...

윈도우 길이: 1,440 블록 (~2시간, 5초/블록 기준)
윈도우 번호 (에포크 내): window_index = ((block_height - 1) % 17280) / 1440
  → 에포크 내 0~11 인덱스, 매 에포크 리셋
윈도우/에포크: 12개
```

| 항목 | 값 |
|------|-----|
| `epoch_length` | 17,280 블록 (거버넌스 파라미터) |
| `windows_per_epoch` | 12 (거버넌스 파라미터) |
| `window_length` | 1,440 블록 (`epoch_length / windows_per_epoch`) |
| 에포크 실제 시간 | ~24시간 (5초/블록 기준) |
| 윈도우 실제 시간 | ~2시간 (5초/블록 기준) |
| 에포크 전환 시점 | `block_height % epoch_length == 0` |
| 윈도우 전환 시점 | `block_height % window_length == 0` |

에포크와 윈도우는 **절대적**이다. 블록 높이 1부터 시작하며 고정 간격으로 반복된다. 노드 등록 시점과 무관하다. 에포크 0은 블록 1~17,280, 에포크 1은 블록 17,281~34,560이다. 에포크 경계(`block_height`가 `epoch_length`의 배수)에서 에포크가 전환되며, 경계 블록 자체는 이전 에포크의 마지막 블록이다.

**시간-블록 변환표** (5초/블록 기준):

| 기간 | 블록 수 | 용도 |
|------|---------|------|
| 2시간 (1 윈도우) | 1,440 | 활동 보상 자격 판정 |
| 1일 (1 에포크 = 12 윈도우) | 17,280 | 활동 보상 분배, 수수료 상태 갱신, feegrant period |
| 21일 | 362,880 | 언본딩 기간 |
| 180일 (6개월) | 3,110,400 | feegrant 만료 |
| 1년 (365일) | 6,307,200 | 활동 기록 + HashIndex 프루닝 보존, blocks_per_year (인플레이션 계산) |

**윈도우 경계 이벤트** (매 1,440 블록):

1. 윈도우 완료 이벤트 발생 (`window_completed`). 활동 기록 자체는 MsgSubmitActivity 처리 시 저장

**에포크 전환 이벤트** (매 17,280 블록):

`x/activity` EndBlocker:
1. 활동 자격 집계: 직전 에포크의 윈도우별 활동 제출 현황 확인, 자격 노드 수(N_a) 카운트
2. 수수료 정산: 에포크 동안 수집된 활동 수수료 처리
3. 다음 에포크 수수료 상태 캐싱: 포화율 기반 활동 수수료 및 유효 feegrant 쿼터 계산
4. 활동 기록 프루닝: `activity_pruning_keep_blocks` 이전 기록 삭제 (에포크당 최대 1,000건)

`x/node` EndBlocker:
5. agent_share 변경 적용: 유예 기간(1에포크) 만료된 변경사항 반영

### 보상 공식: 동적 이중 보상 풀

블록 보상은 **위임 풀**과 **활동 풀**로 분배한다. 두 풀의 비율은 활동 노드 수(N_a)와 Active Validator Set 크기(N_d)에 따라 **동적으로 결정**된다.

```
에포크 블록 보상 총합 (R)
├── 위임 풀: R × delegation_ratio
│   → 지분율 비례 분배 (DPoS)
│   → 노드 에포크 보상 = 위임풀총합 × 지분율
│
└── 활동 풀: R × activity_ratio
    → 에포크 전환 시 자격 충족 노드에게 균등 일괄 분배
    → 노드 에포크 활동 보상 = 에포크활동풀총합 ÷ 자격 노드 수 (N_a)
```

#### 동적 비율 공식

```
delegation_ratio = max(D_min, N_d / (N_a + N_d))
activity_ratio   = 1 - delegation_ratio
```

| 변수 | 정의 |
|------|------|
| `N_a` | 활동 자격 충족 노드 수. REGISTERED + ACTIVE 모두 포함. 상한 없음 |
| `N_d` | Active Validator Set 노드 수. max_validators에 의해 상한 고정 (150~200) |
| `D_min` | 위임 풀 최소 비율. 거버넌스 파라미터 (기본 0.3) |
| `지분율` | (자기 스테이킹 + 위임받은 토큰) / 네트워크 총 스테이킹 |
| `활동 자격` | 에포크 내 12윈도우 중 **8윈도우 이상**에서 각 1건 이상 MsgSubmitActivity 제출. REGISTERED 노드도 자격 가능 |
| `min_active_windows` | 8 (거버넌스 파라미터). 에포크 내 활동 보상을 받기 위해 충족해야 하는 최소 윈도우 수 |

**중복 계산**: 활동 자격을 충족한 Active Validator는 N_a와 N_d **양쪽 모두에** 포함된다. 이 노드는 위임 보상 + 활동 보상을 모두 수령한다.

#### 동적 비율의 동작

```
네트워크 성장에 따른 자동 전환:

초기 (N_a ≈ N_d):
  → delegation_ratio = 40~60% (자연 비율)
  → 위임 보상이 크다 → 밸리데이터 부트스트랩 인센티브

성장기 (N_a > N_d):
  → delegation_ratio 자연 하락
  → 활동 노드 집단의 보상 비중 증가

완숙기 (N_a >> N_d):
  → delegation_ratio = D_min (0.3)에 수렴
  → 활동 노드 집단이 최대 70%까지 도달
  → 위임 세력은 30% 보장
```

**Floor 활성화 조건**: N_a > N_d × (1 - D_min) / D_min. D_min = 0.3, N_d = 150일 때 N_a > 350에서 Floor 작동.

#### 설계 의도

활동 노드 수가 증가할수록 활동 노드 집단의 보상 비중이 커진다. 개별 노드 보상은 줄어들지만, 집단 전체의 지분은 확대된다. 이를 통해 대규모 위임자의 영향력을 활동 참여로 견제할 수 있다. 위임 보상은 활동 노드가 적을수록 커지며, D_min이 위임의 최소 보장선을 유지하여 DPoS 합의 보안 인센티브를 보호한다.

**활동 자격 판정 상세**:

```
활동 보상 자격:
  조건: 에포크 내 12윈도우 중 8윈도우 이상에서 각 1건 이상 MsgSubmitActivity 제출
  판정 시점: 에포크 전환 시 직전 에포크의 윈도우별 활동 기록 집계
  보상 분배: 에포크 전환 시 자격 충족 노드에게 일괄 균등 분배
  미충족 시: 에포크 활동 보상 전액 미지급 (8윈도우 미만 활동 = 0 보상)
  최소 제출: 8 TX/에포크
  허용 유휴: 4윈도우 (8시간) — 서버 장애, 네트워크 지연 내성

  인센티브 구조:
    → 8/12 충족: 에포크 활동 풀의 1/N_a 수령
    → 7/12 이하: 활동 보상 0. 위임 보상만 수령 가능
    → All-or-nothing 방식이므로 지속적 활동에 강한 인센티브

  신규 노드: 에포크 중간에 등록한 경우, 남은 윈도우에서 활동 시작
             다음 완전 에포크부터 자격 판정 적용

  윈도우 활동 기록:
    window_activity[node][window_number] = submission_count
    → 에포크 전환 시 12윈도우 중 submission_count ≥ 1인 윈도우 수 ≥ 8이면 자격 충족
```

#### 활동 비용 모델과의 결합

활동 노드가 밸리데이터 수 대비 과포화될 때 **활동 수수료**가 자동으로 부과된다. 포화율 `S = N_a / (N_d × fee_threshold_multiplier)`를 기준으로:

```
S ≤ 1.0 (Bootstrap):  activity_fee = 0, 쿼터 변동 없음
S > 1.0 (Growth):     activity_fee = base_activity_fee × (S - 1)^0.5
                       feegrant_quota 축소 (최소 min_feegrant_quota까지)
```

이 비용은 보상 공식과 결합하여 **자기 조절 균형점**을 형성한다:
- 활동 풀 보상: `R' × activity_ratio / N_a` (N_a 증가 시 감소)
- 활동 비용: `base_activity_fee × (S - 1)^0.5` (N_a 증가 시 증가)
- 균형점: 보상 = 비용이 되는 N_a에서 Sybil 노드 진입 자연 차단

상세 공식과 시뮬레이션은 [Activity Protocol](04_activity_protocol.md)을 참조한다.

#### 노드 내 보상 분배

```
노드 에포크 총 보상 (위임 보상 + 활동 보상)
├── 커미션 (operator가 설정한 커미션율%)
│   ├── operator 몫: (100 - agent_share)%
│   └── agent 몫:    agent_share%          ← 에이전트 지갑으로 자동 분배
└── 위임자: 나머지 (위임 비율에 비례)
```

`agent_share`는 노드 등록 시 operator가 설정한다 (0-100%). 커미션 보상 중 일부를 에이전트 지갑에 자동 분배하여, 에이전트가 위임 없이도 운영 자금을 확보할 수 있다.
